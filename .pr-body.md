## feat(codemode): connector model + durable runtime, snippets, and vite plugin

Redesigns codemode into class-based **connectors** running on a durable **runtime**. The model gets one tool — `codemode({ code })` — and discovery, approvals, replay, and state all live behind it.

Two layers, cleanly split:

- **Executor** — the simple, stateless code sandbox. Runs a block of code once, dispatches tool calls back. `DynamicWorkerExecutor` (Workers), `IframeSandboxExecutor` (browser).
- **Runtime** — public runtime handle plus `CodemodeRuntime` DurableObject facet. The handle owns the executor/connectors for this run; the facet owns durable log, approvals, and snippets.

### Connectors

`WorkerEntrypoint` subclasses that bridge external services. Constructors carry dependencies; a connector is three things — `name()`, optional `instructions()`, and `tools()`. Everything about a tool (docs, schema, approval, execution, revert) lives in one record entry; derived connectors generate the record and you decorate it via the `tool(name, t)` hook:

```ts
// github.codemode.ts
export class GithubConnector extends McpConnector<Env> {
  name() {
    return "github";
  }
  protected instructions() {
    return "Use for GitHub operations.";
  }
  protected createConnection() {
    return this.conn;
  }
  // decorate derived MCP tools: approval + rollback in one place
  protected tool(name, t) {
    if (name === "create_issue") {
      return {
        ...t,
        requiresApproval: true,
        revert: (_args, result) => this.closeIssue(result.number)
      };
    }
    return t;
  }
}
```

**Base classes:** `CodemodeConnector` (author `tools()` directly; AI SDK toolsets are shape-compatible), `McpConnector`, `OpenApiConnector`
**Authoring surface:** `name()`, `instructions()?`, `tools()`, `tool(name, t)?`
**RPC wire surface** (derived, Gatekeeper-compatible): `describe()`, `executeTool()`, `revertAction()`, `getTypeScriptTypes()`

### Runtime — durable execution via abort-and-replay

`CodemodeRuntime` is a DO facet that owns a per-execution **tool-call log** (the replay spine):

- Every connector call **and** every `codemode.step(name, fn)` is recorded in the log.
- Reads / steps execute and their result is recorded.
- An approval-required action is recorded as `pending` and the run **aborts**.
- On resume, the same code re-runs: calls already in the log are served from it (noop — reads/steps return recorded results), and the newly-approved action executes for real. The run proceeds to the next pause or to completion.

```
run 1:  list_prs() ─exec→ [...]          [applied]
        create_issue() ─PAUSE─           [pending]   ✗ aborts
user approves
run 2:  list_prs() ─replay→ [...]        (from log, no re-exec)
        create_issue() ─exec→ { number } (approved, runs for real)   ✓ completes
```

Lifecycle helpers (driven by the agent, e.g. from `@callable()` methods):

```ts
const runtime = createCodemodeRuntime({ ctx, connectors, executor });
const pending = await runtime.pending(); // list actions awaiting approval
await runtime.approve(); // approve + continue
await runtime.approve({ executionId }); // resume a specific execution
await runtime.reject({ seq }); // reject — ends execution
await runtime.rollback(); // revert applied actions in reverse
const runs = await runtime.executions(); // audit trail, newest first
await runtime.saveSnippet("name", { executionId }); // promote a run to a snippet
await runtime.snippets(); // list; runtime.deleteSnippet(name) removes
```

### codemode.step — the explicit side-effect boundary

A closure can't cross the RPC boundary, so `step` is **sandbox-side JS** (a provider `prelude`) that runs the closure locally and records only its result via two host primitives. Anything that isn't a connector call — raw `fetch`, random, time — goes in a step so it runs once and replays thereafter:

```ts
const id = await codemode.step("gen-id", () => crypto.randomUUID());
const data = await codemode.step("fetch", async () =>
  (await fetch(url)).json()
);
```

Steps share the same sequence/cursor as tool calls, so they interleave and replay in order. Replay divergence (the Nth recorded op differs across runs) is a hard error.

### Model-facing tool

One tool, one input: `codemode({ code })`. Inside the sandbox:

```ts
codemode.search("pull request")     // ranked search across methods + snippets
codemode.describe("github.foo")     // TypeScript type docs
codemode.step(name, fn)             // record-once side-effect boundary
codemode.run("name", input)         // run a snippet the developer saved
github.list_pull_requests(...)      // connector methods as globals
```

### Architecture

Connectors are passed as **env bindings** to the sandbox; calls go via Workers RPC directly (no ToolDispatcher serialization), through the runtime for a replay/execute/pause decision:

```
sandbox: github.list_pull_requests(args)
  → env.__connectors.github.callTool("list_pull_requests", args)
  → runtime.decide(...) → execute → GithubConnector.executeTool() → runtime.recordResult(...)
```

The platform SDK (`codemode` namespace) uses the provider/dispatcher path since it's host-created, not a WorkerEntrypoint.

```
Agent DO
  ├─ runtime handle    (executor, connectors, lifecycle helpers)
  └─ facet: codemode   (CodemodeRuntime — durable log, approvals, state)

runtime.tool()       (spawns the facet, builds platform SDK, runs code in the Executor)
```

### Snippets

Reusable code patterns. Connectors provide raw capability; snippets are recipes that worked. The model writes and runs scripts; the developer promotes the good ones; the model reuses them:

```ts
// host side — e.g. a "save this script" button
const runs = await runtime.executions();
await runtime.saveSnippet("repo-overview", {
  executionId: runs[0].id,
  description: "Fetch repo metadata, open PRs, and latest releases."
});

// sandbox side — the model
const overview = await codemode.run("repo-overview", { owner, repo });
```

Snippets live on the runtime facet, surface in `codemode.search`/`describe`, and the facet's identity is derived from the connector set — so a snippet can never reference a connector that isn't present. There is no separate skill-source interface; external snippet sources (e.g. Think) can come later.

### Vite plugin

`@cloudflare/codemode/vite` discovers `*.codemode.ts` files and auto-exports connector classes (and `CodemodeRuntime`) from the worker entry:

```ts
// vite.config.ts
import codemode from "@cloudflare/codemode/vite";
export default { plugins: [codemode()] };

// server.ts
import { GithubConnector } from "./github.codemode" with { type: "connectors" };
```

### Other changes

- Executor-style ranked search (normalize, tokenize, score, coverage threshold, cap 50)
- `ToolAnnotations` type (`requiresApproval`) — annotate only what needs a human; unannotated methods run immediately. The approval UI describes the action from the method's own descriptor plus its args
- `DynamicWorkerExecutor.bindings` slot for passing ServiceStubs as env
- `ConnectorBinding` on `Executor.execute()` options; `ResolvedProvider.prelude` for sandbox-side fns
- `OpenApiConnector` exposes two overridable primitives — `spec()` (the OpenAPI doc, returned into the sandbox, no prompt tokens) and `request()` (an authenticated request). The model reads the spec and writes its own lookup; no pre-baked per-operation methods. (`request`, not `fetch` — `fetch` is reserved by `WorkerEntrypoint`.)
- Example: `examples/codemode-connectors` (`github.codemode.ts`, `repoapi.codemode.ts`)
- Docs: `docs/codemode/{index,connectors,runtime,snippets,approvals,vite-plugin}.md`

### Design notes

- **Abort-and-replay, not generators.** True mid-stack suspension would require `yield*` everywhere (a real error surface for LLM-written code). We keep plain async/await and re-run between pauses — cheap, since prior calls serve from the log. `step` gives explicit-effect correctness, the one idea worth borrowing from generator-based durable-execution engines.
