## feat(codemode): connector model + durable runtime, skills, and vite plugin

Redesigns codemode into class-based **connectors** running on a durable **runtime**. The model gets one tool — `codemode({ code })` — and discovery, approvals, replay, and state all live behind it.

Two layers, cleanly split:

- **Executor** — the dumb code sandbox. Runs a block of code once, dispatches tool calls back. `DynamicWorkerExecutor` (Workers), `IframeSandboxExecutor` (browser).
- **Runtime** — `CodemodeRuntime`, a DurableObject facet that wraps an executor and makes execution durable via **abort-and-replay**.

### Connectors

`WorkerEntrypoint` subclasses that bridge external services. Constructors carry dependencies; identity/behavior come from overridable template methods:

```ts
// github.codemode.ts
export class GithubConnector extends McpConnector<Env> {
  name() { return "github"; }
  protected instructions() { return "Use for GitHub operations."; }
  protected createConnection() { return this.conn; }
  annotations() {
    return {
      list_pull_requests: { observation: true },
      create_issue: { requiresApproval: true, approvalDescription: "Create a GitHub issue" },
    };
  }
  // optional: enables rollback
  async revertAction(method, args, result) {
    if (method === "create_issue") await this.closeIssue(result.number);
  }
}
```

**Base classes:** `CodemodeConnector`, `McpConnector`, `OpenApiConnector`, `ToolsetConnector`
**RPC surface** (Gatekeeper-compatible): `describe()`, `callTool()`, `getTypeScriptTypes()`
**Template methods:** `name()`, `instructions()`, `annotations()`, `loadDescriptors()`, `executeTool()`, optional `revertAction()`

### Runtime — durable execution via abort-and-replay

`CodemodeRuntime` is a DO facet that owns a per-execution **tool-call log** (the replay spine):

- Every connector call **and** every `codemode.step(name, fn)` is recorded in the log.
- Observations / steps execute and their result is recorded.
- An approval-required action is recorded as `pending` and the run **aborts**.
- On resume, the same code re-runs: calls already in the log are served from it (noop — observations/steps return recorded results), and the newly-approved action executes for real. The run proceeds to the next pause or to completion.

```
run 1:  list_prs() ─exec→ [...]          [applied]
        create_issue() ─PAUSE─           [pending]   ✗ aborts
user approves
run 2:  list_prs() ─replay→ [...]        (from log, no re-exec)
        create_issue() ─exec→ { number } (approved, runs for real)   ✓ completes
```

Lifecycle helpers (driven by the agent, e.g. from `@callable()` methods):

```ts
await resumeCodemode({ ctx, connectors, executor });              // approve + continue
await resumeCodemode({ ctx, connectors, executor, executionId }); // resume a specific branch
await rejectCodemode(ctx, seq);                                   // reject — ends execution
await rollbackCodemode({ ctx, connectors });                      // revert applied actions in reverse
const forkId = await forkCodemode(ctx);                           // snapshot into an independent branch
```

### codemode.step — the explicit side-effect boundary

A closure can't cross the RPC boundary, so `step` is **sandbox-side JS** (a provider `prelude`) that runs the closure locally and records only its result via two host primitives. Anything that isn't a connector call — raw `fetch`, random, time — goes in a step so it runs once and replays thereafter:

```ts
const id   = await codemode.step("gen-id", () => crypto.randomUUID());
const data = await codemode.step("fetch", async () => (await fetch(url)).json());
```

Steps share the same sequence/cursor as tool calls, so they interleave and replay in order. Replay divergence (the Nth recorded op differs across runs) is a hard error.

### Model-facing tool

One tool, one input: `codemode({ code })`. Inside the sandbox:

```ts
codemode.search("pull request")     // ranked search across methods + skills
codemode.describe("github.foo")     // TypeScript type docs
codemode.connectors()               // list available connectors
codemode.pending()                  // actions awaiting approval
codemode.step(name, fn)             // record-once side-effect boundary
codemode.fork()                     // checkpoint / branch
codemode.get(key) / codemode.set(key, value)   // per-execution scratchpad
codemode.run("skill-name", input)   // execute a reusable skill
github.list_pull_requests(...)      // connector methods as globals
```

### Architecture

Connectors are passed as **env bindings** to the sandbox; calls go via Workers RPC directly (no ToolDispatcher serialization), through the runtime for a replay/execute/pause decision:

```
sandbox: github.list_pull_requests(args)
  → env.__connectors.github.callTool("list_pull_requests", args)
  → runtime.decide(...) → execute → GithubConnector.executeTool() → runtime.recordResult(...)
```

The platform SDK (`codemode` namespace) uses the provider/dispatcher path since it's host-created, not a WorkerEntrypoint.

```
Agent DO
  ├─ facet: codemode   (CodemodeRuntime — durable log, approvals, state)
  └─ createProxyTool   (spawns the facet, builds platform SDK, runs code in the Executor)
```

### Skills

Reusable code patterns. Connectors provide raw capability; skills provide recipes. Codemode only defines the interface — any source (Think, bundled, R2) implements it:

```ts
interface CodemodeSkillSource {
  id: string;
  list(): Promise<CodemodeSkill[]>;
  load?(name: string): Promise<CodemodeSkill | null>;
}

createProxyTool({ ctx, executor, connectors: [github, repoApi], skills: [bundledSkills] });
```

### Vite plugin

`@cloudflare/codemode/vite` discovers `*.codemode.ts` files and auto-exports connector classes (and `CodemodeRuntime`) from the worker entry:

```ts
// vite.config.ts
import codemode from "@cloudflare/codemode/vite";
export default { plugins: [codemode()] };

// server.ts
import { GithubConnector } from "./github.codemode" with { type: "connectors" };
```

### Design notes

- **Abort-and-replay, not generators.** True mid-stack suspension would require `yield*` everywhere (a real error surface for LLM-written code). We keep plain async/await and re-run between pauses — cheap, since prior calls serve from the log. `step` gives explicit-effect correctness and `fork` gives checkpoints, the two ideas worth borrowing from generator-based durable-execution engines.
