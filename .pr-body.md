## feat(codemode): connector model + durable runtime, snippets, and vite plugin

Redesigns codemode into class-based **connectors** running on a durable **runtime**. The model gets one tool ‚Äî `codemode({ code })` ‚Äî and discovery, approvals, replay, and state all live behind it.

Two layers, cleanly split:

- **Executor** ‚Äî the simple, stateless code sandbox. Runs a block of code once, dispatches tool calls back. `DynamicWorkerExecutor` (Workers), `IframeSandboxExecutor` (browser).
- **Runtime** ‚Äî public runtime handle plus `CodemodeRuntime` DurableObject facet. The handle owns the executor/connectors for this run; the facet owns durable log, approvals, snippets, and scratch state.

### Connectors

`WorkerEntrypoint` subclasses that bridge external services. Constructors carry dependencies; identity/behavior come from overridable template methods:

```ts
// github.codemode.ts
export class GithubConnector extends McpConnector<Env> {
  name() {
    return "github";
  }
  protected instructions() {
    return "Use for GitHub operations.";
  }
  protected createConnection() {
    return this.conn;
  }
  annotations() {
    return {
      list_pull_requests: { observation: true },
      create_issue: {
        requiresApproval: true,
        approvalDescription: "Create a GitHub issue"
      }
    };
  }
  // optional: enables rollback
  async revertAction(method, args, result) {
    if (method === "create_issue") await this.closeIssue(result.number);
  }
}
```

**Base classes:** `CodemodeConnector`, `McpConnector`, `OpenApiConnector`, `ToolsetConnector`
**RPC surface** (Gatekeeper-compatible): `describe()`, `callTool()`, `getTypeScriptTypes()`
**Template methods:** `name()`, `instructions()`, `annotations()`, `loadDescriptors()`, `executeTool()`, optional `revertAction()`

### Runtime ‚Äî durable execution via abort-and-replay

`CodemodeRuntime` is a DO facet that owns a per-execution **tool-call log** (the replay spine):

- Every connector call **and** every `codemode.step(name, fn)` is recorded in the log.
- Observations / steps execute and their result is recorded.
- An approval-required action is recorded as `pending` and the run **aborts**.
- On resume, the same code re-runs: calls already in the log are served from it (noop ‚Äî observations/steps return recorded results), and the newly-approved action executes for real. The run proceeds to the next pause or to completion.

```
run 1:  list_prs() ‚îÄexec‚Üí [...]          [applied]
        create_issue() ‚îÄPAUSE‚îÄ           [pending]   ‚úó aborts
user approves
run 2:  list_prs() ‚îÄreplay‚Üí [...]        (from log, no re-exec)
        create_issue() ‚îÄexec‚Üí { number } (approved, runs for real)   ‚úì completes
```

Lifecycle helpers (driven by the agent, e.g. from `@callable()` methods):

```ts
const runtime = createCodemodeRuntime({ ctx, connectors, executor });
const pending = await runtime.pending(); // list actions awaiting approval
await runtime.approve(); // approve + continue
await runtime.approve({ executionId }); // resume a specific branch
await runtime.reject({ seq }); // reject ‚Äî ends execution
await runtime.rollback(); // revert applied actions in reverse
const forkId = await runtime.fork(); // snapshot into an independent branch
```

### codemode.step ‚Äî the explicit side-effect boundary

A closure can't cross the RPC boundary, so `step` is **sandbox-side JS** (a provider `prelude`) that runs the closure locally and records only its result via two host primitives. Anything that isn't a connector call ‚Äî raw `fetch`, random, time ‚Äî goes in a step so it runs once and replays thereafter:

```ts
const id = await codemode.step("gen-id", () => crypto.randomUUID());
const data = await codemode.step("fetch", async () =>
  (await fetch(url)).json()
);
```

Steps share the same sequence/cursor as tool calls, so they interleave and replay in order. Replay divergence (the Nth recorded op differs across runs) is a hard error.

### Model-facing tool

One tool, one input: `codemode({ code })`. Inside the sandbox:

```ts
codemode.search("pull request")     // ranked search across methods + snippets
codemode.describe("github.foo")     // TypeScript type docs
codemode.connectors()               // list available connectors
codemode.pending()                  // actions awaiting approval
codemode.step(name, fn)             // record-once side-effect boundary
codemode.fork()                     // checkpoint / branch
codemode.get(key) / codemode.set(key, value)   // per-execution scratchpad
codemode.save("name", { description })        // save the script as a snippet
codemode.run("name", input)         // run a saved snippet
github.list_pull_requests(...)      // connector methods as globals
```

### Architecture

Connectors are passed as **env bindings** to the sandbox; calls go via Workers RPC directly (no ToolDispatcher serialization), through the runtime for a replay/execute/pause decision:

```
sandbox: github.list_pull_requests(args)
  ‚Üí env.__connectors.github.callTool("list_pull_requests", args)
  ‚Üí runtime.decide(...) ‚Üí execute ‚Üí GithubConnector.executeTool() ‚Üí runtime.recordResult(...)
```

The platform SDK (`codemode` namespace) uses the provider/dispatcher path since it's host-created, not a WorkerEntrypoint.

```
Agent DO
  ‚îú‚îÄ runtime handle    (executor, connectors, lifecycle helpers)
  ‚îî‚îÄ facet: codemode   (CodemodeRuntime ‚Äî durable log, approvals, state)

runtime.tool()       (spawns the facet, builds platform SDK, runs code in the Executor)
```

### Snippets

Reusable code patterns. Connectors provide raw capability; snippets are recipes that worked. The model writes a script, promotes it with `codemode.save(name, { description })`, and re-runs it with `codemode.run(name, input)`:

```ts
await codemode.save("repo-overview", {
  description: "Fetch repo metadata, open PRs, and latest releases."
});

const overview = await codemode.run("repo-overview", { owner, repo });
```

Snippets live on the runtime facet, surface in `codemode.search`/`describe`/`snippets()`, and the facet's identity is derived from the connector set ‚Äî so a snippet can never reference a connector that isn't present. There is no separate skill-source interface; external snippet sources (e.g. Think) can come later.

### Vite plugin

`@cloudflare/codemode/vite` discovers `*.codemode.ts` files and auto-exports connector classes (and `CodemodeRuntime`) from the worker entry:

```ts
// vite.config.ts
import codemode from "@cloudflare/codemode/vite";
export default { plugins: [codemode()] };

// server.ts
import { GithubConnector } from "./github.codemode" with { type: "connectors" };
```

### Other changes

- Executor-style ranked search (normalize, tokenize, score, coverage threshold, cap 50)
- `ToolAnnotations` type (`observation`, `requiresApproval`, `approvalDescription`)
- `DynamicWorkerExecutor.bindings` slot for passing ServiceStubs as env
- `ConnectorBinding` on `Executor.execute()` options; `ResolvedProvider.prelude` for sandbox-side fns
- Per-execution scratchpad state (forks get an independent copy)
- `OpenApiConnector` exposes two overridable primitives ‚Äî `spec()` (the OpenAPI doc, returned into the sandbox, no prompt tokens) and `request()` (an authenticated request). The model reads the spec and writes its own lookup; no pre-baked per-operation methods. (`request`, not `fetch` ‚Äî `fetch` is reserved by `WorkerEntrypoint`.)
- Example: `examples/codemode-connectors` (`github.codemode.ts`, `repoapi.codemode.ts`)
- Docs: `docs/codemode/{index,connectors,runtime,snippets,search-and-describe,approvals,vite-plugin}.md`

### Design notes

- **Abort-and-replay, not generators.** True mid-stack suspension would require `yield*` everywhere (a real error surface for LLM-written code). We keep plain async/await and re-run between pauses ‚Äî cheap, since prior calls serve from the log. `step` gives explicit-effect correctness and `fork` gives checkpoints, the two ideas worth borrowing from generator-based durable-execution engines.
